# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation — Auth, Roles & Project Skeleton.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope:** Login screen, Register screen, Home page (post-login skeleton), protected /admin placeholder,
> auth error states, loading/disabled states during auth requests.
> These primitives (tokens, typography, spacing, color, button/input/form, app shell) become
> the reusable foundation all 9 phases build on.

---

## Design System

| Property         | Value                                                                 |
|------------------|-----------------------------------------------------------------------|
| Tool             | none — plain Tailwind CSS utilities (no shadcn/radix; greenfield)    |
| Preset           | not applicable                                                        |
| Component library| none — hand-rolled from Tailwind CSS 4 utility classes               |
| Icon library     | none for Phase 1 — inline SVG for role badge icon if needed          |
| Font             | system-ui stack: `ui-sans-serif, system-ui, sans-serif` (no web font fetch; fast cold-start) |

**Rationale (source: RESEARCH.md Standard Stack):**
Tailwind CSS 4.3.0 is locked in RESEARCH.md as the styling approach. `components.json` is absent
(confirmed: greenfield project, no frontend directory yet). A custom component library would add
unnecessary complexity for a learning-oriented walking skeleton. Plain Tailwind utilities with a
handful of hand-rolled components (`Button`, `Input`, `FormField`, `Badge`, `Card`) are sufficient
for the 4 screens in this phase and give later phases a zero-opinion base to extend.

## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage                                           |
|-------|-------|-------------------------------------------------|
| xs    | 4px   | Icon gaps, label-to-input vertical spacing      |
| sm    | 8px   | Input internal padding (horizontal), button gaps |
| md    | 16px  | Default element spacing, card inner padding     |
| lg    | 24px  | Section gap inside auth form card               |
| xl    | 32px  | Gap between form card and page edges (mobile)   |
| 2xl   | 48px  | Vertical centering offset / top breathing room  |
| 3xl   | 64px  | Not used in Phase 1 screens                     |

Exceptions:
- Touch targets on the Submit/Logout button: minimum height 44px (accessibility — WCAG 2.5.5)
- Auth form card max-width: 400px (fixed, not a spacing token)

## Typography

| Role    | Size  | Weight          | Line Height | Usage                                              |
|---------|-------|-----------------|-------------|----------------------------------------------------|
| Body    | 16px  | 400 (regular)   | 1.5         | Form labels, help text, paragraph copy             |
| Label   | 14px  | 500 (medium)    | 1.4         | Input field labels, badge text, nav items          |
| Heading | 20px  | 600 (semibold)  | 1.2         | Auth form card title ("Đăng nhập", "Đăng ký")     |
| Display | 28px  | 700 (bold)      | 1.15        | Home page greeting "Xin chào {username}"           |

**Weights in use:** 400 (regular), 600 (semibold). Weight 500 (medium) is used only for Label
role — acceptable as an intermediate; the two "primary" weights for the design system are
400 and 600. Weight 700 is reserved exclusively for the Display greeting on the Home page.

**Font stack (source: Design System table above):**
```css
font-family: ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
```
No Google Fonts or web font fetches — keeps the walking skeleton fast and offline-capable.

## Component Inventory

Hand-rolled components to be created in `frontend/src/components/` during Phase 1.
These become the reusable primitives for all later phases.

### `<Button>`

| Variant     | Appearance                                          | Usage                              |
|-------------|-----------------------------------------------------|------------------------------------|
| primary     | `bg-blue-600 text-white hover:bg-blue-700`          | Submit (login, register)           |
| destructive | `text-red-600 hover:bg-red-50 border border-red-200`| Logout button                      |
| secondary   | `bg-gray-100 text-gray-700 hover:bg-gray-200`       | Cancel / secondary actions         |

States: default, hover, focus (ring-2 ring-blue-600 ring-offset-2), disabled (opacity-60 + cursor-not-allowed), loading (spinner icon + "..." label, button disabled).

Height: minimum 44px. Padding: `px-4 py-2.5` (10px vertical = 44px total with 16px font + border).

### `<Input>`

States: default (`border-gray-300`), focus (`border-blue-600 ring-1 ring-blue-600`), error (`border-red-500 ring-1 ring-red-500`), disabled (`bg-gray-50 cursor-not-allowed`).

Sizing: `w-full px-3 py-2 text-base` (16px text prevents iOS auto-zoom).

Password field: include a show/hide toggle button (inline icon button, no additional package).

### `<FormField>`

Wraps label + input + error message. Error message: `text-sm text-red-600 mt-1`.

### `<Badge>` (role badge on Home page)

Variants: `admin` (amber) and `user` (emerald) per Color section above.
Size: `text-xs font-medium px-2.5 py-0.5 rounded-full border`.

### `<AuthCard>` (layout wrapper for Login / Register forms)

Centered card: `max-w-[400px] w-full mx-auto bg-white rounded-xl shadow-md p-8`.
Page centering: `min-h-screen flex items-center justify-center bg-gray-100 px-4`.

### `<AppShell>` (layout for Home / Admin pages)

Slim top navigation bar: `bg-white border-b border-gray-200 px-6 py-3 flex items-center justify-between`.
Contains: app name left-aligned, right side: username + role badge + Logout button.
Content area: `max-w-3xl mx-auto px-4 py-8`.

## Screens & Interaction Contract

### Login Page (`/login`)

**Layout:** AuthCard centered on gray-100 background.

**Elements:**
- Heading: "Đăng nhập" (20px semibold)
- Input: username or email (label: "Tên đăng nhập hoặc email")
- Input: password with show/hide toggle (label: "Mật khẩu")
- Submit Button (primary): "Đăng nhập"
- Error message area (below form, above submit): shown when API returns error
- Footer link: "Chưa có tài khoản? Đăng ký" → navigates to `/register`

**States:**
- Idle: form empty, submit enabled
- Loading: submit button shows spinner + disabled, inputs disabled
- Error: error message shown below password field (generic, no field identification)
- Success: redirected to `/` (home)

**Redirect behavior:** Already-authenticated user visiting `/login` → redirect to `/`.

---

### Register Page (`/register`)

**Layout:** AuthCard centered on gray-100 background.

**Elements:**
- Heading: "Đăng ký" (20px semibold)
- Input: username (label: "Tên đăng nhập")
- Input: email (label: "Email")
- Input: password with show/hide toggle (label: "Mật khẩu")
- Submit Button (primary): "Đăng ký"
- Error message area: shown when API returns validation error
- Footer link: "Đã có tài khoản? Đăng nhập" → navigates to `/login`

**States:** same pattern as Login (idle / loading / error / success).

**Success:** redirect to `/login` with a brief success indicator ("Đăng ký thành công — hãy đăng nhập").
*(Registration does NOT auto-login: simplest and most secure pattern for Phase 1.)*

**Nav bar elements:**
- Left: app name "VDT WebRTC" (16px semibold, text-gray-800)

**Content area elements:**

- Spacer

  (14px, text-gray-400, italic — visual signal this is a skeleton)

**Source:** D-08 in CONTEXT.md.

### Admin Page (`/admin`) — Protected Placeholder

**Layout:** AppShell (same nav bar as Home).

**Content area elements:**
- Heading: "Trang quản trị" (20px semibold)
- Body: "Chỉ Admin mới có thể truy cập trang này." (16px regular, text-gray-600)

**Access control (UI route guard):** `<ProtectedRoute requiredRole="ADMIN">` wrapping this route.

**Source:** D-08 in CONTEXT.md; specifics section in CONTEXT.md.

### Route Guard — Unauthorized Access

When a User role attempts to access `/admin`:
- Silent redirect to `/` (no error page, no toast — just redirect)
- No 403 error copy shown in Phase 1 (route guard redirects before API call)

**Error copy policy (source: CONTEXT.md Claude's Discretion):**
Login errors MUST be generic ("sai tài khoản hoặc mật khẩu") — do NOT identify which field
is wrong. This prevents username enumeration. Applies to both the HTTP 401 and 404 case from
the server (server should also return generic 401 for both).

**Destructive actions in Phase 1:** Logout only.
- No confirmation dialog required (session loss is recoverable — user can re-login)
- Destructive button style (text-red-600 border) provides visual affordance without a modal
- Logout is immediate on click: clear access token + POST /api/auth/logout + redirect to /login

## Interaction Patterns

### Form Submission Flow

1. User submits form
2. Client validates required fields (non-empty); show inline error if empty — do NOT show error before first submit attempt
3. Button transitions to loading state: spinner + disabled text + inputs disabled
4. API call completes:
   - Success → navigate to target page
   - Failure (4xx/5xx) → button returns to default state, error message appears above submit button
5. Error message clears when user starts typing in any field

### Token Persistence (AUTH-02)

On successful login:
- Store `accessToken` in module-level JS variable (NOT localStorage — D-03)
- On page refresh: call `/api/auth/refresh` (httpOnly cookie auto-sent) to rehydrate access token
- While refresh call is in flight: show loading spinner centered on page (not a full-screen block)
- If refresh fails (cookie expired/revoked): redirect to `/login` silently

### Logout Flow

1. User clicks "Đăng xuất"
2. POST `/api/auth/logout` (best-effort — do NOT block on failure)
3. Clear in-memory `accessToken` regardless of API response
4. Redirect to `/login`

### Protected Route Loading State

When `ProtectedRoute` is checking auth (initial load, refresh in-flight):
- Show centered loading indicator (`<div>Đang tải...</div>` with basic spinner)
- Do NOT flash the login page then redirect (prevents FOUC)

## Tailwind CSS 4 Configuration Notes

Tailwind CSS 4 uses CSS-first configuration (no `tailwind.config.js` needed for basic setup).
Import via:
```css
/* src/index.css */
@import "tailwindcss";
```
And in `vite.config.ts`, use `@tailwindcss/vite` plugin (already in RESEARCH.md install commands).

Custom CSS variables for design tokens (define in `src/index.css`):
```css
:root {
  --color-accent: #2563eb;       /* blue-600 */
  --color-destructive: #dc2626;  /* red-600 */
  --color-surface: #ffffff;
  --color-canvas: #f3f4f6;       /* gray-100 */
}
```

These four variables map exactly to the Color contract above and can be extended in later phases
without breaking Phase 1 components.

## Accessibility Baseline

- All form inputs MUST have a visible `<label>` (not placeholder-only)
- Error messages MUST be associated via `aria-describedby` on the input
- Loading states: `aria-busy="true"` on the form or button during submission
- Role badges: `role="status"` or plain text (not aria-hidden — screen readers should announce role)
- Focus order: top-to-bottom, no focus traps outside modal (no modals in Phase 1)
- Color contrast: blue-600 on white = 4.5:1 (AA), red-600 on white = 4.63:1 (AA) — meets WCAG 2.1 AA
