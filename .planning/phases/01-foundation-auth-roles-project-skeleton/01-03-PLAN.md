<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| httpOnly cookie → POST /api/auth/refresh | Token value never accessible to JS; sent automatically by browser |
| Refresh token hash in DB → AuthService | SHA-256 hash in `token_hash` column; raw value only in cookie |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Repudiation | Refresh token reuse after rotation | mitigate | After rotation, old token's `revoked=true` in DB. `findByTokenHashAndRevokedFalse` rejects the old token. Attempted reuse → 401. (D-02 server-side tracking) |
| T-03-02 | Elevation of Privilege | Admin-locked user bypassing via existing refresh token | mitigate | `user.isLocked()` check on every `/api/auth/refresh` call — locked user cannot obtain a new access token even with a valid (non-expired) refresh cookie. Access token TTL is max 15 min (D-03). |
| T-03-03 | Denial of Service | Concurrent 401 storm causing multiple rotation calls | mitigate | `isRefreshing + failedQueue` pattern ensures exactly one refresh call per concurrency window. Without this, parallel requests each rotate the token and the second rotation uses an already-rotated (invalid) token → 401 loop. |
| T-03-04 | Information Disclosure | Refresh token raw value exposure | mitigate | Raw token only in httpOnly cookie — never in response body, never in JS variables. Only SHA-256 hash stored in DB. If DB is compromised, hashes cannot be reversed to raw tokens (pre-image resistant). |
| T-03-05 | Tampering | Expired refresh token accepted | mitigate | `stored.getExpiresAt().isBefore(Instant.now())` check in `AuthService.refresh()` throws `AccessDeniedException` for expired tokens. 7-day TTL balances UX and security. |
</threat_model>
