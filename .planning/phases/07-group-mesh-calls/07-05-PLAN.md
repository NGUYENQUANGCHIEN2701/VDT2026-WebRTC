<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Manual room demo for ADV-03 success criteria and 1-1 smoke</name>
  <files></files>

  <action>
    Perform the real user verification against the scaled backend from Phase 6.

    1. Start the backend topology used for multi-instance signaling:
       `docker compose up --build -d postgres redis rabbitmq coturn backend-1 backend-2 nginx`

    2. Start the frontend if it is not already provided by compose:
       `cd frontend && npm run dev`

    3. Prepare 5 user sessions in separate browser contexts.

    4. Verify success criterion #1:
       - User A starts a group call from the online-user list (D-01) and invites Users B, C, and D.
       - B/C/D accept and join.
       - Confirm each participant sees and hears all others.
       - Confirm the room uses the separate `/group-call` route and the grid reflects 4 users.

    5. Verify success criterion #2:
       - User E attempts to join the already-full room through a legitimate stale invite or a
         test-only WebSocket `join-room` attempt against the full room. Do not add a room-link
         UX just to create this condition; the normal UI still caps initiator selection at 3
         invitees, and automated backend tests remain the primary cap proof.
       - Confirm the join is rejected server-side and the UI shows the approved room-full message.

    6. Verify success criterion #3:
       - Have one participant leave cleanly; confirm only that tile disappears and the other
         peers stay connected.
       - Rejoin a participant, then close one browser tab abruptly to simulate a drop; confirm
         only that peer shows reconnecting/failed and the rest of the room continues.

    7. Verify success criterion #4:
       - Open the DebugPanel with 3 or 4 participants.
       - Confirm the active `maxBitrate` value is visible for peers and corresponds to the mesh cap.
       - Drop back to 2 participants and confirm the cap row returns to uncapped/blank behavior.

    8. Verify cleanup:
       - After everyone leaves, inspect Redis and confirm the room keys are gone.
       - Prefer `SCAN`/targeted lookup over broad destructive inspection. Example:
         `docker compose exec redis redis-cli SCAN 0 MATCH "room:*" COUNT 100`
         `docker compose exec redis redis-cli SCAN 0 MATCH "user-room:*" COUNT 100`
       - The phase is not done if orphan room keys remain after the room is empty.

    9. Final 1-1 smoke:
       - Start a normal 1-1 call between two users.
       - Confirm `/call` still works and the group changes did not break the core product value.
  </action>
