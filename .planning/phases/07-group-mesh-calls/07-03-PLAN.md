## Phase Goal

**As a** frontend engineer building the mesh core, **I want** a browser-side orchestration layer
that can manage multiple peer connections without polluting the 1-1 path, **so that** the UI can
later render rooms while the existing call flow stays stable.

<objective>
Implement the non-UI frontend core for Phase 7: the additive `PeerManager` seam, a new
`MeshManager`, a separate `roomStore`, and the room signaling bridge in `roomActions` /
`messages.ts` / `wsClient.ts`. This plan turns the RED unit tests GREEN and creates the end-to-end
room control plane needed by the UI plan, while deliberately avoiding a refactor of the 1-1
`callStore` or `callActions`.

This plan must implement D-04 joiner-initiates and D-06 dynamic bitrate caps exactly as locked,
and it must keep all non-serializable browser objects out of Zustand.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser room orchestration -> Zustand | Only derived room state crosses into the store; raw WebRTC objects remain outside React state to avoid corruption and accidental leaks. |
| Server room messages -> roomActions | roomActions must treat server messages as authoritative for membership and roster changes; the frontend must not invent its own cap or presence truth. |
