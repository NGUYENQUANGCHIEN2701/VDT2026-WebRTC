# Phase 7 — UI Design Contract: Group Mesh Calls

> Visual and interaction contract for Phase 7 frontend work.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Design approach:** This project uses vanilla CSS custom properties + React inline styles.
> No Tailwind, no shadcn, no component library. All new components in this phase follow the
> established pattern from `components/call/*` and `components/presence/*`.

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, tile status dot gaps, inline badge padding |
| sm | 8px | Tile label padding, grid gap between tiles, button gap in control bar |
| md | 16px | Control bar padding, tile inner padding, invite modal content padding |
| lg | 24px | Invite modal card padding, section spacing in room lobby |
| xl | 32px | Page-level vertical margins |
| 2xl | 48px | Major layout breaks |
| 3xl | 64px | (available, not expected to be needed in this phase) |

Exceptions:
- Grid tile gap: 8px (sm) between tiles in the 2x2 layout (dense, media-first layout)
- Self-view tile is the same grid cell size as remote tiles (no PiP exception — group uses uniform grid, not PiP)
- Touch targets for round control buttons: 56px diameter (inherited from Phase 4 `round()` helper)
- DebugToggle button: 44px × 44px (inherited from Phase 4)
- Multi-select checkboxes in the invite user list: 20px × 20px (minimum accessible touch target is 44px via the li wrapper)

## Screen Inventory

### Screen 1: Home Page — Multi-select Invite Entry (MODIFY OnlineUsersList / OnlineUserRow)

Entry point: user taps/clicks a new "Gọi nhóm" button on the Home page, then selects 1–3 other
online users from the existing list to invite.

**Layout changes to HomePage / OnlineUsersList:**
- Add a "Gọi nhóm" button above or at the top of the OnlineUsersList panel.
  Size: height 44px, min-width 140px, border-radius 8px, border: 1px solid `var(--accent-border)`,
  color: `var(--accent)`, background: transparent. Disabled and opacity 0.4 when a call is already
  active (matches existing "Gọi" button disabled behavior).
- When "Gọi nhóm" mode is active, each online ONLINE user row shows a checkbox (20px) to its left.
  Self and IN_CALL users are non-selectable (dimmed, cursor not-allowed).
- Selected count shown: small label "Đã chọn: N/3" in 14px `var(--text)`, top-right of the list.
- Confirm button: "Mời N người" (accent background, white text, height 44px) appears at the bottom
  of the list when ≥1 user is selected. Disabled (opacity 0.4) if 0 users selected.
- Cancel / exit multi-select: "Huỷ" text button at top-right of multi-select header strip.

**Constraints:**
- Maximum 3 selectable invitees (cap: 4 total including initiator). The 4th checkbox is
  visually disabled once 3 are already selected.
- If the user is already IN_CALL (1-1), "Gọi nhóm" button is hidden (group is a parallel,
  additive flow — do not escalate 1-1 to group; D-01 from CONTEXT.md).

### Screen 2: Outgoing Group Invite (initiator side — new state)

Shown after the initiator taps "Mời N người". Replaces the existing SelfViewPreview for group.

**Layout:**
- Full-screen overlay (fixed inset 0, `rgba(0,0,0,0.5)` backdrop).
- Centered card: background `var(--code-bg)`, border-radius 12px, padding 24px, max-width 400px,
  box-shadow `var(--shadow)`.
- Heading (20px, 600): "Đang mời vào phòng nhóm…"
- Invitee list: each row shows username + status dot (pending / accepted / rejected / timeout).
  Font 16px, gap 8px per row.
- Status dot colors: pending `#6b7280`, accepted `#16a34a`, rejected `#dc2626`.
- Self-view: NOT shown in the invite-pending state (group start is lobby, not preview — no media
  acquired until first invitee accepts and room is confirmed active).
- Cancel button: "Huỷ mời" — full width, height 44px, background `var(--border)`,
  color `var(--text-h)`, border-radius 8px. Appears below the invitee list.

---

### Screen 3: Incoming Group Invite (invitee side — new variant of IncomingCallCard)

Shown when a server `room-invite` message arrives for a user not in a call.

**Layout:**
- Same overlay pattern as IncomingCallCard: fixed inset, `rgba(0,0,0,0.5)` backdrop.
- Centered card: background `var(--code-bg)`, border-radius 12px, padding 24px, max-width 360px.
- Heading (20px, 600): "{initiatorUsername}" (name of who is calling)
- Subheading (16px, `var(--text)`): "Đang mời bạn vào cuộc gọi nhóm" (N người, including self)
  e.g., "Đang mời bạn vào cuộc gọi nhóm (2 người)"
- Two buttons side by side (gap 16px, height 44px, min-width 120px, border-radius 8px):
  - Accept: background `#16a34a`, color `#fff`, label "Tham gia"
  - Reject: background `#dc2626`, color `#fff`, label "Từ chối"
- Auto-timeout: if no action taken in ~30s, card dismisses and invite is treated as declined.
  No visible countdown timer (keep it simple; consistent with 1-1 ring timeout approach).

**If the user is already in a call (1-1 or group):** Incoming group invite is silently discarded
server-side (busy). This card never appears for a busy user.

### Screen 4: Room Full Rejection (new state)

Shown when the server rejects a join attempt because the room is at capacity (5th+ person).

**Layout:**
- Toast-style notice (not a full modal): slides in from top, auto-dismisses after 4s.
- Background `var(--code-bg)`, border-radius 8px, padding `12px 16px`, box-shadow `var(--shadow)`.
- Icon: ⚠ (Unicode, 16px) + text "Phòng đã đầy (tối đa 4 người)" — 14px, `var(--text-h)`.
- Width: max-content, max-width 320px, centered horizontally.

Reuse the existing Toaster component pattern (already in `components/Toaster.tsx`).

### Screen 5: Group Call Page — 2×2 Grid (NEW GroupCallPage or new state in CallPage)

The primary in-call view for group calls. Implemented as a separate `/group-call` route to
protect the existing 1-1 `/call` route (D-02 from CONTEXT.md).

**Overall layout:**
- Full-screen, `background: #000` (video-first, matches 1-1 CallPage outer background).
- Top bar: 44px height, padding 0 16px, flex, space-between.
  Left: room participant count badge "N người" (14px, `var(--text)` on `var(--code-bg)`, border-radius 4px, padding 2px 8px).
  Center: call duration (14px, 600, tabular-nums, `var(--text)`).
  Right: DebugToggle button (44×44px, same as 1-1 CallPage).
- Grid area: flex 1, display grid, grid-template-columns per participant count:
  - 2 participants: `1fr 1fr` (two landscape tiles side by side)
  - 3 participants: `1fr 1fr` top row (2 tiles) + 1 centered tile bottom row
    (use `grid-template-areas` or CSS subgrid to center the lone bottom tile)
  - 4 participants: `1fr 1fr` / `1fr 1fr` (true 2×2)
  Gap: 8px. Padding: 8px.
- Control bar: height 80px (fits round buttons), display flex, justify-content center,
  gap 16px, padding 0 16px. Background: `var(--bg)` (matches 1-1 pattern).
- Debug panel: shown below control bar when open (same DebugPanel component, extended for
  per-peer stats — see Component Inventory section).

**Grid tile structure (one tile per participant including self):**
- `position: relative`, `border-radius: 8px`, `overflow: hidden`, `background: #000`.
- `<video>` element: width 100%, height 100%, object-fit cover.
- Self-view tile: `transform: scaleX(-1)` (mirror, matches Phase 4 self-view).
- Name label overlay: `position: absolute`, bottom 8px, left 8px. Background:
  `rgba(0,0,0,0.55)`, border-radius 4px, padding `2px 6px`, font 14px 600, color `#fff`.
  Content: username + "(bạn)" suffix for self tile.
- Mic-muted indicator: small 🔇 overlay, position absolute top 8px left 8px, font 14px.
  Shown when the peer's mic is muted.
- Cam-off state: video hidden; tile shows `background: #1f2937`; centered avatar circle
  (96px, border-radius 50%, `background: var(--code-bg)`) with initial character (40px, `#fff`).
  Matches RemoteCamOffOverlay.
- Connection state overlay (per-tile partial-mesh failure surface, from D-05 CONTEXT.md):
  - `reconnecting`: semi-transparent overlay `rgba(0,0,0,0.45)` + centered spinner
    (36px, `border: 3px solid rgba(255,255,255,0.3)`, `border-top-color: #fff`) +
    text "Đang kết nối lại…" (15px 600 `#fff`). Matches 1-1 reconnecting overlay exactly.
  - `failed`: solid overlay `rgba(31,41,55,0.9)` + centered text "Kết nối thất bại" (15px 600
    `#fff`) + smaller sub-text "Đã ngắt kết nối với người này" (13px `rgba(255,255,255,0.7)`).
  - `connected`: no overlay.
  - `connecting`: same as reconnecting but text "Đang kết nối…".

**Control bar buttons (group call):**
- Mute mic: round 56px, same as Phase 4 MuteButton. Active: `#6b7280`; inactive: `var(--code-bg)`.
- Toggle cam: round 56px, same as Phase 4 CamToggleButton.
- Leave room: round 56px, `background: #dc2626`, color `#fff`, label: ✕ (no "hang up" —
  user leaves, room may continue). aria-label: "Rời phòng".

## Interaction Contracts

### IC-1: Multi-select Invite Flow

1. User taps "Gọi nhóm" on HomePage. OnlineUsersList enters multi-select mode.
2. Checkboxes appear left of each ONLINE (non-self, non-IN_CALL) user row.
3. User checks 1–3 users. "Đã chọn: N/3" counter updates.
4. After 3 selected, remaining unselected ONLINE users show checkbox disabled (opacity 0.4,
   cursor not-allowed). Already-selected users remain unchecked.
5. "Mời N người" button activates. Tapping it sends `create-room` → server creates room,
   fans out invites.
6. UI transitions to OutgoingGroupInviteCard overlay showing invitee statuses.
7. "Huỷ mời" cancels the room creation and returns to normal Home view.

### IC-2: Invite Accept / Reject Flow

1. Invitee receives `room-invite` server message → GroupInviteModal appears.
2. Tapping "Tham gia": sends `join-room` intent → server Lua atomic cap check.
   - On success: server returns current member list → GroupCallPage opens, MeshManager
     creates one PeerManager per existing member → offers sent (joiner-initiates, D-04).
   - On room-full: GroupInviteModal dismissed, room-full toast shown.
3. Tapping "Từ chối": sends `decline-room-invite` → modal dismissed, user stays on Home.
4. No response in ~30s: invite auto-expires server-side; GroupInviteModal dismissed.

### IC-3: Grid Tile Dynamic Layout

The grid re-computes CSS on roster changes without page reload:
- 2 participants: 2 columns, 1 row. Each tile: 50% width.
- 3 participants: 2 columns top row (2 tiles), 1 centered tile spanning full width at bottom
  (use `grid-column: 1 / -1` on the lone tile, set `justify-self: center`, `max-width: 50%`).
- 4 participants: 2×2. Each tile: 50% width × 50% height.
Transition: CSS `transition: grid-template-columns 0.2s ease` on the grid container.

### IC-4: Partial Mesh Failure Surfacing (D-05)

When a specific peer's RTCPeerConnection enters `disconnected` or `failed` state:
- Only that peer's tile shows the reconnecting/failed overlay.
- All other tiles remain unaffected.
- PeerManager attempts ICE restart (existing behavior) — tile shows "Đang kết nối lại…"
  during restart attempt.
- If ICE restart fails after the established timeout, tile shows "Kết nối thất bại" overlay.
- The failed tile remains in the grid (tombstone tile — do not remove it; the peer may still
  be in the room from the server's perspective until they leave/timeout).
- A participant leaving cleanly (leave-room intent) removes their tile from the grid.

### IC-5: Per-Sender Bitrate Cap (D-06)

Bitrate caps apply dynamically as roster crosses thresholds:
- 2 participants (1-1): no cap.
- 3–4 participants: cap each RTCRtpSender's video track to 350 kbps
  (`RTCRtpSender.setParameters({ encodings: [{ maxBitrate: 350_000 }] })`).
  Value 350 kbps chosen as midpoint of the 300–500 kbps range in D-06; adjust after
  empirical testing (researcher-recommended default).
- Cap is applied/removed dynamically: when the 3rd peer joins, all senders cap; when
  the room drops to 2, caps are removed.
- Verifiable in DebugPanel: a new row "maxBitrate" shows the active cap value (or "—" if uncapped).

### IC-6: Leave Room Behavior

- Any participant (including initiator) can leave at any time with the ✕ LeaveRoomButton.
- On leave: send `leave-room` → server broadcasts `participant-left` → all remaining peers
  receive the event → each closes only that peer's PeerManager tile → grid reflows.
- Room is deleted server-side when the last member leaves.
- If the local user is now alone after others leave (not a self-initiated leave):
  "Tất cả đã rời phòng" overlay shown on GroupCallPage → auto-navigate to Home after 2s.
- On leaving, GroupCallPage unmounts → MeshManager.destroy() closes all PeerConnections,
  releases media tracks, clears roomStore.

### IC-7: Debug Panel in Group Mode

The existing DebugPanel is extended to handle group calls:
- In group mode, DebugPanel shows stats per peer (tabbed or stacked, one section per peer).
- Each section header: peer's username, bold, 14px.
- Per-peer rows: same as 1-1 (Codec, Bitrate, Độ phân giải, ICE, RTT, Mất gói).
- Additional row per peer: "maxBitrate" — shows active cap value ("350 kbps") or "—" (uncapped).
- DebugToggle behavior unchanged: single button in GroupCallPage top bar.

## Accessibility Checklist

- All interactive tiles with connection-state overlays include `role="status" aria-live="polite"`
  on the overlay element.
- GroupInviteModal: `role="dialog" aria-modal="true" aria-labelledby` (matches IncomingCallCard).
- LeaveRoomButton: `aria-label="Rời phòng"` (action described, not icon described).
- Checkbox list in multi-select: `<ul role="list">` with `<li>` wrapping each checkbox row;
  each checkbox `<input type="checkbox">` has an associated `<label>`.
- Participant count badge: `aria-label="N người trong phòng"` (not just "N người").
- Self tile video element: `aria-label="Camera của bạn"` (matches 1-1 pattern).
- Remote tile video elements: `aria-label="Camera của {username}"` (matches 1-1 pattern).
- Color is never the sole indicator: tile connection state uses both color overlay AND text.
- Spinner in reconnecting overlay: `aria-hidden="true"` (matches 1-1 spinner usage).
