## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated WebSocket session -> room service | Room membership, inviter identity, and per-user routing must be derived server-side from the session principal, never from client-supplied identity fields. |
| Backend -> Redis room keys | Room membership and cleanup are cross-instance shared state; orphaned keys or non-atomic joins become correctness and access-control bugs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Elevation of Privilege | join-room flow | mitigate | Server-side join checks and invite/room membership validation stay in RoomService; do not trust client-side selection or route claims. |
| T-07-05 | Tampering | room cap | mitigate | Lua-based atomic cap from D-03 blocks concurrent 5th joins and becomes a blocking test gate. |
| T-07-06 | Information Disclosure | room existence/full responses | mitigate | Return only the approved room-full / generic failure signals; do not expose extra membership details to unauthorized users. |
| T-07-07 | Denial of Service | orphan room keys | mitigate | leave_room.lua and disconnect cleanup are blocking; tests must prove empty-room deletion and reverse-key cleanup. |
</threat_model>
