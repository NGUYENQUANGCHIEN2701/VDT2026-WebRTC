<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| GitHub Actions runner -> repository secrets | CI jobs run with GitHub-provided ephemeral credentials only; no project secrets (JWT_SECRET, DB passwords) are referenced by these 3 jobs since none of them start the full application against real config |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-06 | Information Disclosure | CI workflow accidentally referencing real `.env` secrets | high | mitigate | This plan's 3 jobs (backend/frontend/docker-build) never source `.env` or reference `${{ secrets.* }}` for application credentials — `mvn verify` uses Testcontainers' own ephemeral containers with test-only defaults, `docker build` builds images without runtime env injection. No real secret can leak because none is referenced. |
| T-09-07 | Tampering | Unpinned `uses:` action versions | low | accept | `actions/checkout@v4`, `actions/setup-java@v4`, `actions/setup-node@v4` are pinned to major version tags (not `@main`/`@latest`), consistent with standard GitHub Actions security guidance; full SHA-pinning is not required at this project's risk level (internal learning project, no untrusted external contributors triggering workflow runs with secrets) |

</threat_model>
