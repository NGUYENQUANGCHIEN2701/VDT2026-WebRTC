## Phase Goal

**As a** developer running the compose stack, **I want to** start two backend instances behind an
nginx load balancer so the cross-instance routing is proven in a real compose demo where two browser
tabs (on different instances) complete a call.

<objective>
Compose + nginx slice: the infrastructure layer that enables the live manual demo of SCAL-01.

Creates: nginx/conf.d/vdt.conf (WebSocket-aware round-robin LB).
Modifies: docker-compose.yml — splits old single backend into backend-1 + backend-2 (each with
INSTANCE_ID env var), adds nginx service, keeps all existing services (postgres, redis, rabbitmq,
coturn) unchanged.

No code changes — this plan is infrastructure only. The Redis pub/sub routing implemented in
Plan 02 handles instance affinity; nginx uses plain round-robin with no sticky sessions per D-07.

Deferred (not in this plan): frontend service, Prometheus, Grafana (Phase 9, INFR-02).

Output: docker compose up (backend-1/backend-2/nginx/redis/postgres/rabbitmq/coturn) starts
cleanly; nginx exposes port 8080 and routes /ws and /api to the two backends.
</objective>

    Create directory nginx/conf.d/ if it does not exist, then create nginx/conf.d/vdt.conf.

    The config must contain exactly these sections (per RESEARCH.md Pattern 10):

    Section 1 — map for WebSocket upgrade header:
    - map $http_upgrade $connection_upgrade: default upgrade; '' close

    Section 2 — upstream backend:
    - server backend-1:8080
    - server backend-2:8080
    - keepalive 32
    - NO ip_hash directive (per D-07 — sticky sessions explicitly rejected; Redis routing handles affinity)

    Section 3 — server block listen 80:
    - location /ws: proxy_pass http://backend; proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection $connection_upgrade; proxy_set_header Host $host; proxy_read_timeout 3600s; proxy_send_timeout 3600s
    - location /api: proxy_pass http://backend; proxy_http_version 1.1; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for

    Rationale for proxy_read_timeout 3600s: prevents nginx from closing idle WS connections (nginx default is 60s; heartbeat is 25s but idle connections may exceed 60s during backgrounded tabs).
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-07 | Elevation of Privilege | nginx as new external entry point | mitigate | nginx exposes only /ws and /api locations; no root location or directory listing; backend ports not exposed to host; compose network provides layer isolation |
| T-06-08 | Information Disclosure | Backend health endpoint via nginx | accept | /actuator/health is not routed through nginx (/api location only routes /api prefix); healthcheck uses direct localhost:8080 within container; no external exposure of actuator |
| T-06-09 | Tampering | X-Forwarded-For header injection | accept | nginx sets X-Forwarded-For; backend does not currently use it for auth decisions; no impact on security model; acceptable at demo scale |
| T-06-10 | Denial of Service | nginx single point of failure | accept | Demo scope; Phase 9 adds healthchecks; for Phase 6 demo the nginx failure is visible in compose logs |
| T-06-SC | Tampering | No new npm/pip/cargo installs | accept | No package manager installs; nginx:1.27-alpine is an official Docker Hub image — no slopcheck needed; image tag is pinned per CLAUDE.md |
</threat_model>
