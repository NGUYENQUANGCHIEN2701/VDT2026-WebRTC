<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Backend → Redis | State machine writes; must use StringRedisTemplate to avoid key serialization mismatch |
| Lua script → Redis | Atomic execution; KEYS and ARGV must be in correct order to prevent callId confusion |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | Spoofing | create_call.lua KEYS ordering | mitigate | KEYS[2]=caller, KEYS[3]=callee is explicit in script comments; transition_call.lua validates current state before writing — wrong-order KEYS would cause wrong busy check, caught by CallLifecycleTest.busy_rejection |
| T-04-02-02 | Tampering | Redis state leak on server restart | mitigate | All call:{callId} and user-call:{userId} keys have TTL=300s (5 min) set in create_call.lua ARGV[5]; timer Futures are lost on restart but Redis TTL auto-expires leaked state |
| T-04-02-03 | Tampering | Java serialization prefix on Redis keys (Pitfall 4) | mitigate | Use StringRedisTemplate exclusively; never inject RedisTemplate<Object,Object> into call services |
| T-04-02-SC | Tampering | npm/pip/cargo installs | accept | Only one new Maven artifact (spring-boot-starter-data-redis) — official Spring Boot artifact, no slopcheck needed |
</threat_model>
