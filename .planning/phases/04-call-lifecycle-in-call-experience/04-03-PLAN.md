<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → WS handler | Untrusted intent frames; identity must be bound from session principal |
| WS handler → CallService | Validated intents only; callId existence enforced by CAS |
| CallService → Redis | Atomic CAS; no transition without matching from-state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Spoofing | intent `from`/body fields | mitigate | PresenceWebSocketHandler reads username from session.getAttributes(); CallService handlers take the authenticated principal as first param and derive callerId/calleeId from the Redis record, never the body — verified by CallLifecycleTest using real mintToken JWTs |
| T-04-03-02 | Tampering | unauthorized transition (third user injects accept/hangup) | mitigate | CallService validates the principal is a participant of call:{callId} (callerId or calleeId in the Redis hash) before any transition; non-participants get no broadcast and CAS is not run |
| T-04-03-03 | Tampering | double-accept / simultaneous hangup race | mitigate | All transitions go through transition_call.lua CAS — only one transition from a given from-state succeeds; second attempt returns 0 and is ignored (CallStateMachineTest.concurrent_transition proves this) |
| T-04-03-04 | DoS | glare exploitation (forced simultaneous calls) | mitigate | create_call.lua is atomic; at most one call per user-pair; user-call: pointer TTL 300s prevents permanent busy lock (D-03 deterministic resolution) |
| T-04-03-05 | DoS | grace-period abuse (reconnect loop to hold call open) | accept | Grace timer is server-owned and reset per reconnect, but Redis TTL 300s on call keys is the hard ceiling; acceptable for single-instance demo scope |
| T-04-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages in this plan |
</threat_model>
