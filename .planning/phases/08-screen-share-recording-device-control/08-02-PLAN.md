## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client screen share request â†’ browser | getDisplayMedia requires user activation; the implementation must not cache or reuse display streams across calls. |
| recording-state relay â†’ backend | areActiveCallPeers must be the sole gate for relay; no client-supplied `from` field is trusted. |
| Device switch â†’ getUserMedia | exact deviceId constraints can fail; the implementation must not tear down the call on failure and must keep the previous track active. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-08-04 | Spoofing | recording-state relay | mitigate | areActiveCallPeers validates call membership before relaying; username comes from the server-side session, not client body. |
| T-08-05 | Denial of Service | device switch mid-call | mitigate | On getUserMedia failure, previous track remains active and the call is not disrupted. |
| T-08-06 | Elevation of Privilege | RecordingStateRelay to third party | mitigate | `to` field is validated against the known peer; relay to arbitrary users is rejected. |
</threat_model>
