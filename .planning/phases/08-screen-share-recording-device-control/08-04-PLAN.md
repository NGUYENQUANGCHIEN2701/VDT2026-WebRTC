  <name>Task 2: State preservation edge cases</name>
  <files>
    frontend/src/realtime/callActions.ts,

  </files>

  <action>
    Verify and harden the following state preservation behaviors. Read the existing Wave 2
    implementations and patch them if the edge cases are not already handled.

    **Camera-off preservation after screen share stops:**
    In `stopScreenShare()` and `stopRoomScreenShare()`:
    - After acquiring the restore camera track, explicitly set:
      `cameraTrack.enabled = !useCallStore.getState().camOff`
      (or roomStore equivalent for group).
    - The restored track must inherit the pre-share camera-off state.
    - `setCamOff(true)` should NOT be called again — the store already holds the correct value.
    - Relay `media-state` after stopping so the remote party sees the correct cam-off state.

    **Mute preservation after microphone switch:**
    In `switchMicrophone()` and `switchRoomMicrophone()`:
    - After acquiring the new audio track, set `newTrack.enabled = !micMuted` from the store.
    - Replace the sender THEN stop the old track.
    - Do NOT call `setMicMuted` (store value unchanged — preserving the mute state IS correct).

    **Screen share should override camera-off temporarily:**
    From D-09/decision and UI-SPEC: starting screen share explicitly turns video on even if
    camera was off. Verify that `startScreenShare()`:
    - Sets `camOff(false)` in the store after replacing the sender.
    - Relays `media-state` with `camOff: false` to the remote.
    - Restores the ORIGINAL `camOff` value when screen share stops (not the post-share value).

    Store the original `camOff` value in `cameraTrackBeforeShare` module scope or add a
    separate `camOffBeforeShare` module-scope variable. On `stopScreenShare`, restore both
    the camera track and the `camOff` value:
    ```ts
    if (camOffBeforeShare) {
        cameraTrack.enabled = false
        useCallStore.getState().setCamOff(true)
    }
    // relay media-state with restored values
    ```

    **Switching camera while screen sharing:**
    In `switchCamera()`: if `isScreenSharing` is true, update `selectedCameraDeviceId` in the
    store but skip the `replaceTrack` step entirely. The new camera will be acquired in
    `stopScreenShare()` via `selectedCameraDeviceId`.

    Apply the same guard in `switchRoomCamera()`.
  </action>

  </verify>

  <done>
    - stopScreenShare restores exact pre-share camOff value including relay.
    - switchMicrophone preserves micMuted without calling setMicMuted.
    - startScreenShare stores original camOff before overriding.
    - switchCamera while screen sharing only updates the selected device ID without replacing track.
    - TypeScript clean.
  </done>

    **MediaRecorder unsupported guard** in `MorePanel.tsx`:
    ```ts
    const recorderSupported = typeof MediaRecorder !== 'undefined'
    ```
    When false: disable "Start recording" button and show helper text:
    "Recording is not supported in this browser. The call can continue."

    **No remote stream guard** in `MorePanel.tsx`:
    The More panel cannot directly access the `PeerManager`. Use a prop or a store flag.
    Add `remoteStreamReady: boolean` to `callStore` (set to `true` in the existing
    `bumpRemoteStream` call, false on reset) OR pass `remoteStreamReady` as a prop from
    `CallPage.tsx` (preferred to avoid store coupling for this UI detail).
    When `remoteStreamReady` is false: disable Start recording with helper text:
    "Recording starts after the call media is connected."

    **MediaRecorder runtime error handling** in `RecordingController`:
    Add an `onerror` handler on the recorder:
    ```ts
    recorder.onerror = () => {
        this._isRecording = false
        cancelAnimationFrame(this._rafId)
        this._audioCtx?.close()
        // Callback or store update to signal error
    }
    ```
    The controller should accept an optional `onError?: (msg: string) => void` callback.
    In CallPage.tsx pass: `onError: (msg) => useCallStore.getState().setRecordingError(msg)`.
    Show error via `recordingError` store field when set: a destructive notice in the HUD
    "Recording stopped due to an error." with auto-dismiss after 5s.

    **Empty chunks after stop** in `RecordingController.stop()`:
    If the collected chunks array is empty or total size is 0: return `null` instead of a
    preview URL. CallPage should handle `null` result by showing "No recording data was
    captured." as a toast (not a modal).

    **Speaker selector entirely hidden** in `MorePanel.tsx`:
    Verify the conditional rendering uses:
    ```ts
    const supportsSinkId = 'setSinkId' in HTMLMediaElement.prototype
    ```
    The block must be `{supportsSinkId && <section>...</section>}` — a hidden or disabled
    select is NOT acceptable per UI-SPEC D-10.
  </action>
