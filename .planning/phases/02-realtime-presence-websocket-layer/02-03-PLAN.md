<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser WS client → /ws | Client sends the in-memory access token in the connect URL; the server (Plan 02) is the gate. Client renders only server-pushed snapshots — it never asserts presence. |
| server snapshot → rendered UI | The client replaces its list wholesale from each snapshot (no client-side authority over who is online — D-03 full-snapshot). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02 | Spoofing | client-rendered identity | mitigate | UI derives the list and self-exclusion from server snapshots + `useAuthStore` user; the client never trusts a self-asserted `from`. Server owns identity (Plan 02). |
| T-02-03 | Information Disclosure | token in WS connect URL | accept | Same short-lived in-memory access token as Phase 1 (never persisted); known accepted risk, hardened in v2 (STAB-05). Client does not log the URL. |
| T-02-07 | Denial of Service | client reconnect storm after backend restart | mitigate | Exponential backoff capped ~30s + random jitter (Pitfall 4); no reconnect after a kick (kicked-flag guard) so the superseded tab does not fight the new session. |
| T-02-08 | Tampering | kick vs transient-drop confusion | mitigate | Client distinguishes `session-superseded` (notice + redirect, no reconnect) from a plain close (reconnect, dim list) by the control message — D-02 / UI-SPEC Interaction Patterns. |
| T-02-SC | Tampering | npm install (frontend) | accept | Zero new runtime npm dependencies this plan (native WebSocket + vanilla CSS per CLAUDE.md / UI-SPEC Registry Safety); only Vitest (dev) was added in Plan 01. No [ASSUMED]/[SUS] package — no blocking checkpoint required. |
</threat_model>
