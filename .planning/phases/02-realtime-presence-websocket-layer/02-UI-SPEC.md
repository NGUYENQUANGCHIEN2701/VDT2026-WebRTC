# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Realtime Presence & WebSocket Layer.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Scope (small realtime UI surface — NOT a design-system phase):**
> 1. Online-users list rendered on the Home page after login (username + status badge, realtime).
> 2. WebSocket connection-state indicator (connecting / connected / reconnecting).
> 3. Single-session "kick" notice + redirect to `/login` when the session is superseded (D-02).
> 4. Empty state (no other users online) and loading state (while the WS connects).
>
> This phase adds the FIRST realtime surface on top of the Phase 1 home page. It introduces no new
> screens — only new elements inside the existing `HomePage`, plus one notice surface for the kick.

## Design System

| Property | Value |
|----------|-------|
| Tool | **none** — plain vanilla CSS + inline styles (matches the implemented baseline) |
| Preset | not applicable — `components.json` absent; shadcn not initialized |
| Component library | **none** — hand-rolled inline-styled React elements (no radix / base-ui) |

**Detected baseline (source of truth — codebase scan, 2026-06-14):**
- `frontend/package.json` — **no** Tailwind, shadcn, radix, lucide, or any component-library dependency.
- `frontend/src/pages/HomePage.tsx` and `LoginPage.tsx` — implemented with **inline `style={{}}` objects**.
- `frontend/src/index.css` — defines reusable CSS custom properties (`--accent`, `--text`, `--text-h`,
  `--border`, `--bg`, `--shadow`, `--sans`) and a `prefers-color-scheme: dark` block.

**Decision (ASSUMPTION — noted because no interactive Q&A in this run):** Phase 2 stays on the
**existing vanilla-CSS / inline-style baseline**. This is a learning project; introducing Tailwind or a
component library here would be scope creep and contradicts the implemented code. The Phase 1 UI-SPEC
*mentioned* Tailwind, but the shipped frontend does not use it — this contract aligns with the **shipped reality**.
Reuse the `index.css` CSS variables for all colours so light/dark mode keeps working for free.

## Typography

Inherits the Phase 1 system (2 weights total: 400 regular + 500/600 semibold). Phase 2 adds **no new sizes** beyond this set.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display | 28px | 600 | 1.2 | Existing home greeting "Xin chào, {username}" (unchanged) |

**Font:** inherit `var(--sans)` from `:root`. No new font fetch.

## Color

All colours reference the **existing `index.css` CSS variables** so dark mode continues to work. New
semantic colours for presence status are added as a small, fixed palette (light/dark pairs).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--bg)` (`#fff` light / `#16171d` dark) | Page background, list-row background |
| Secondary (30%) | `var(--social-bg)` / `var(--code-bg)` | Online-list row hover/zebra background, notice card surface |
| Accent (10%) | `var(--accent)` (`#aa3bff` / `#c084fc`) | Online-list section title underline/marker; "connected" indicator text — **reserved list below** |
| Destructive | `#dc2626` (light) / `#f87171` (dark) | Kick-notice emphasis; "reconnecting"/disconnected indicator |

**Accent reserved for (explicit list — never "all interactive elements"):**
- The online-list section heading marker (small accent dot or left-border on the section title).
- The "Đã kết nối" (connected) connection-state indicator text/dot.
- That is all — list rows, usernames, and badges do NOT use accent.

**Presence status colours (semantic palette — fixed, not part of the 60/30/10 budget):**

| Status | Dot colour (light / dark) | Badge text / bg (light) | Meaning |
|--------|---------------------------|--------------------------|---------|
| ONLINE | `#16a34a` / `#4ade80` (green) | `#166534` on `#dcfce7` | User is connected, available |
| IN_CALL | `#d97706` / `#fbbf24` (amber) | `#92400e` on `#fef3c7` | Forward-compatible — **rendered only from Phase 4**; Phase 2 never emits it but the badge component MUST handle the enum value |

**Connection-state indicator colours:**

| WS state | Colour (light / dark) | Dot |
|----------|------------------------|-----|
| connecting | `#6b7280` / `#9ca3af` (grey) | pulsing/solid grey |
| connected | `var(--accent)` or green `#16a34a` | solid |
| reconnecting / disconnected | `#dc2626` / `#f87171` (destructive) | solid/blinking |

**Contrast note:** green `#166534` on `#dcfce7` = ~7:1 (AA+); amber `#92400e` on `#fef3c7` = ~6:1 (AA).
Both meet WCAG 2.1 AA. In dark mode, use the lighter dot colours listed above on the dark `--bg`.

## Component Inventory

Hand-rolled, inline-styled (or a co-located `.css`) React elements. No package. Place under
`frontend/src/components/presence/` (new) per the Phase 1 feature-package convention.

### `<ConnectionIndicator>`
A small pill/row near the top of the home content showing WS state.
- Layout: `[dot] [text]`, horizontal, `gap: 4px`, `font: 14px/1.4 weight 600`.
- States drive dot colour + text (see Copywriting + Color). Always rendered (even while connecting).
- `role="status"` + `aria-live="polite"` so screen readers announce reconnect transitions.

### `<StatusBadge status>`
Pill rendered per online-list row.
- `padding: 4px 8px; border-radius: 999px; font: 14px/1 weight 600;`
- Accepts the full enum `ONLINE | IN_CALL` (forward-compatible per D-03). Phase 2 only ever receives
  `ONLINE`; the `IN_CALL` branch MUST exist and compile but is exercised in Phase 4.
- Colour by status from the Presence status colours table.

### `<OnlineUserRow user>`
One row in the list.
- Layout: `[status dot] [username (flex:1)] [StatusBadge]`, `padding: 8px 16px`, `align-items: center`,
  `gap: 8px`, bottom border `1px solid var(--border)`.
- Hover background: `var(--social-bg)`.
- Username: 16px/400, colour `var(--text-h)`.
- Self is **excluded** from the list (the snapshot may include self; the UI filters out the current user).

### `<OnlineUsersList>`
The section container mounted in `HomePage`.
- Section title (Heading 20px/600): "Đang trực tuyến ({count})" with a small accent left-marker.
- Renders loading state, empty state, or the list of `<OnlineUserRow>` (see Screens).
- `max-width: 480px`. Rows in a simple vertical stack.

### `<SessionKickNotice>`
Full-area (or centered card) notice shown when `session-superseded` arrives, then auto-redirect.
- Centered card on the existing page: `var(--social-bg)`/`var(--code-bg)` surface, `var(--shadow)`,
  `border-radius: 8px`, `padding: 24px`, `max-width: 400px`, text-align center.
- Body copy (16px/400) + a manual "Đăng nhập lại" link/button as a fallback if auto-redirect is blocked.
- Redirect to `/login` automatically after ~2s (see Interaction Patterns).

## Screens & Interaction Contract

### Home Page (`/`) — Online Users surface

**Layout:** Existing `HomePage` (left-aligned content, `padding: 24px`). The greeting, role, and Đăng xuất
button stay as-is. The new presence surface is **inserted below** the existing greeting block, in this order:

1. `<ConnectionIndicator>` (small, directly under the greeting/role line)
2. `<OnlineUsersList>` section (title + list / states)

**Visual entry point:** the section heading "Đang trực tuyến ({n})" is the anchor; the eye flows
greeting → connection indicator → list heading → rows. The accent left-marker on the heading draws the eye.

**States of `<OnlineUsersList>`:**

| State | Trigger | What the user sees |
|-------|---------|--------------------|
| Loading | WS connecting, no snapshot yet | Heading "Đang trực tuyến" (no count) + body text "Đang kết nối..." (14px, `var(--text)`) |
| Empty | Snapshot received, 0 other users online | Heading "Đang trực tuyến (0)" + centered muted message (see Copywriting), 32px top padding |
| Populated | Snapshot with ≥1 other user | Heading with live count + stack of `<OnlineUserRow>` |
| Reconnecting | WS drops then retries | Keep the LAST known list visible but dim it (`opacity: 0.6`); ConnectionIndicator shows "Đang kết nối lại..." Do NOT clear the list to empty on a transient blip. |

**Realtime behavior (D-03 — full snapshot):**
- The server pushes a **full snapshot** on every join/leave/status change. The UI **replaces** its list
  state with each snapshot (no client-side delta merging).
- The list MUST visibly update **without a page refresh** — this is a hard success criterion (PRES-01).
- The count in the heading updates with every snapshot.
- No enter/leave animation is required; a simple instant re-render is acceptable for the demo. (Optional:
  a 150ms fade is allowed but not mandated — keep it lightweight.)

**Self-exclusion:** the current user is filtered out of the rendered list (compare against
`useAuthStore.getState().user.username`). Heading count reflects **other** users only? — **No:** to match
the snapshot semantics, the heading count reflects the number of rows actually rendered (others). Keep
count and visible rows consistent.

---

### Single-Session Kick (D-02, PRES-03)

**Trigger:** the old tab's WebSocket receives a `session-superseded` control message, after which the
server closes the socket.

**Behavior:**
1. On receiving the control message, the client immediately stops heartbeats and renders `<SessionKickNotice>`
   (replacing the home content or as a centered overlay card).
2. Clear the in-memory access token (the session is dead) so the app can't keep making authed calls.
3. Auto-redirect to `/login` after **~2 seconds** (long enough to read the notice; D-02 requires a
   notice, not a silent disconnect).
4. The `/login` page shows the kick reason as a one-line info banner (see Copywriting) so the user
   understands why they were logged out.

**Important:** this is **notice + redirect**, never a silent disconnect (D-02 is explicit). If the WS
simply closes without the control message (e.g. network drop), that is a *reconnect* case, NOT a kick —
the client must distinguish the two by the presence of the `session-superseded` message before closing.

**Primary CTA for this phase:** there is **no new primary action button** — Phase 2 is a passive/observe
surface (the list and indicators auto-update). The only interactive element introduced is the kick-notice
**"Đăng nhập lại"** fallback link. (Calling a user is Phase 3 — explicitly out of scope here.)

**Error-state policy:** WS errors are shown via the `<ConnectionIndicator>` (transient) and, only after the
reconnect ceiling is hit, a single actionable line ("Không thể kết nối... Vui lòng tải lại trang."). Do not
spam toasts on every reconnect attempt — the indicator carries reconnect status.

**Destructive actions in Phase 2:** none introduced by this phase. The kick is **server-initiated**, not a
user-triggered destructive action, so it needs a notice (D-02) but **no confirmation dialog**. (Đăng xuất
from Phase 1 is unchanged and still needs no confirmation.)

## Interaction Patterns

### WebSocket lifecycle → UI mapping
1. On Home mount (authenticated): the WS service connects using the in-memory access token
   (`useAuthStore.getState().token`), token transport per researcher/planner discretion (query param vs
   subprotocol). UI shows ConnectionIndicator = "Đang kết nối..." and list = Loading.
2. On open + first snapshot: indicator → "Đã kết nối", list → Populated/Empty.
3. On each subsequent snapshot: list re-renders (full replace), count updates.
4. On unexpected close (no `session-superseded`): indicator → "Đang kết nối lại...", list dims (opacity 0.6),
   client retries with backoff (backoff details = planner discretion). On success → step 2.
5. On `session-superseded`: render kick notice, clear token, redirect to `/login` after ~2s (D-02).
6. On Home unmount / logout: close the WS cleanly.

### Heartbeat (visual side only)
- Heartbeat cadence (~25s) and offline threshold (~60s) are **D-04 server/protocol concerns**, not visual.
  The UI does not render heartbeat status directly; offline users simply disappear from the next snapshot.

### Realtime update guarantee
- The list MUST update on snapshot push **without any user action or refresh** (PRES-01 success criterion).
  This is the single most important interaction contract of the phase — the checker/auditor should verify it.

## Accessibility Baseline

- `<ConnectionIndicator>`: `role="status" aria-live="polite"` so reconnect/connected transitions are announced.
- `<OnlineUsersList>`: render rows as a `<ul>`/`<li>` list (semantic), heading as a real `<h2>`.
- Status conveyed by **colour + text label** (badge text "Trực tuyến"/"Đang gọi"), never colour alone —
  colour-blind safe.
- Status dot has `aria-hidden="true"` (decorative; the badge text carries the meaning).
- Kick notice: `role="alertdialog"` or `role="alert"` so it is announced immediately; manual "Đăng nhập lại"
  link is keyboard-focusable in case auto-redirect is blocked.
- Contrast: all status/badge pairs meet WCAG 2.1 AA (see Color contrast note). Reuse `index.css` variables
  so dark mode preserves contrast.

## Assumptions (made in lieu of interactive Q&A)

1. **Vanilla CSS / inline styles retained** (no Tailwind, no component library, no icon package) — matches
   the shipped frontend and the learning-project intent. Reuse `index.css` CSS variables.
2. **Presence surface lives inside the existing `HomePage`**, inserted below the greeting — no new route/screen.
3. **Status dot uses a coloured `<span>`/Unicode glyph**, not an icon library.
4. **Kick auto-redirect delay ~2s** with a manual fallback link — long enough to read, short enough to feel responsive.
5. **No enter/leave animations mandated** — instant re-render on snapshot is acceptable for demo scale.
6. **Self is excluded** from the rendered online list.
7. **IN_CALL badge branch is built but dormant** in Phase 2 (wired in Phase 4 per D-03).
