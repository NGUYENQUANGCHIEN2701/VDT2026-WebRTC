## Phase Goal

**As an** admin, **I want to** see a live dashboard of system health and browse all calls across the system, **so that** I can monitor the application during live demos and investigate call history for any user.

Vertical slice 3 of 3: admin dashboard + system-wide call history (ADMN-02, ADMN-03).

This plan introduces CallMetrics (AtomicLong counters with midnight reset), wires the three
counter increments into CallService, extends AdminService with getDashboard + getSystemHistory,
adds GET /api/admin/dashboard + GET /api/admin/history to AdminController, and delivers the
DashboardCards and SystemHistoryTable frontend components integrated into AdminPage.

Covers ADMN-02 (system-wide history) and ADMN-03 (live dashboard) entirely per D-12..D-16.

Purpose: Admins need operational visibility. Dashboard counts refresh every 5 seconds — "live
enough" for a demo without requiring a dedicated WebSocket push channel (D-15). Daily stats reset
at midnight per D-14 using AtomicLong (NOT Micrometer Counter, which has no built-in daily reset —
RESEARCH.md §Pitfall §Pattern 7).

Output: CallMetrics service, DashboardDto, AdminHistoryRow DTO, 2 new admin endpoints, 2 new
frontend components, and AdminPage extended with tabs/sections for Dashboard and System History.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin browser → GET /api/admin/dashboard | Admin JWT required; RBAC enforced at SecurityConfig |
| Admin browser → GET /api/admin/history | Admin JWT required; no per-user scoping needed (admin sees all) |
| Backend → Redis (KEYS scan for active-calls) | Internal network call; Redis not exposed to untrusted callers |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-12 | Information Disclosure | GET /api/admin/dashboard — exposes system-wide call counts | mitigate | Protected by SecurityConfig /api/admin/** hasRole("ADMIN"); regular users get 403. Counts are aggregate (not PII-exposing). |
| T-05-13 | Information Disclosure | GET /api/admin/history — exposes all users' call records | mitigate | Protected by SecurityConfig ADMIN role; only admins can access call history of all users. This is the intended admin capability (ADMN-02). |
| T-05-14 | Denial of Service | StringRedisTemplate.keys("user-call:*") blocking Redis event loop | accept | Acceptable for Phase 5 single-instance demo scale. TODO comment in AdminService.getDashboard: "Phase 6: replace with dedicated Redis counter to avoid O(N) KEYS blocking". Risk is low at demo scale. |
| T-05-15 | Tampering | SQL injection via username filter in admin history | mitigate | CallHistoryRepository.findAllCalls uses @Query with @Param (parameterized JPA query, never string concatenation). Input validated by the JPA layer automatically. |
| T-05-16 | Denial of Service | callMetrics counter lost on service restart | accept | AtomicLong counters reset on restart — acceptable for MVP demo dashboard (RESEARCH.md §Pitfall 5). Documented in CallMetrics as a class-level comment. |
| T-05-17 | Tampering | DashboardCards frontend request interval — client-controlled polling rate | accept | The refetchInterval is set in the frontend code and does not affect server security. The server endpoint is protected by JWT + RBAC regardless of polling frequency. |
| T-05-SC | Tampering | No new npm packages in this plan | accept | No new npm dependencies; all existing packages already audited. |
</threat_model>
