<objective>
## Phase Goal

**As an** admin, **I want to** lock/unlock users and change their roles inline in the user table with a confirmation step, **so that** bad actors are immediately expelled from the system and my own account cannot be accidentally locked.

Vertical slice 2 of 3: admin user management (ADMN-01).

This plan extends AdminService + AdminController with lock/unlock/changeRole operations, wires
the SessionRegistry force-disconnect (D-11), enforces self-protection (D-10) server-side,
adds the ConfirmModal component, and extends AdminPage with the inline Hành động column.

Covers ADMN-01 entirely: lock/unlock, role change, force-disconnect on lock, self-protection.

Purpose: A locked user must lose access in two ways: (1) future login attempts fail (Spring
Security accountLocked — already wired from Phase 1), and (2) any active WS session closes
immediately (SessionRegistry.close). The realtime Phase 4 call path then handles the drop
gracefully (grace→dropped for the surviving peer, D-11).

Output: 3 new backend endpoints, extended AdminService, ConfirmModal component, and extended
AdminPage with full ADMN-01 behavior.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin browser → PATCH /api/admin/users/{id}/lock | Admin JWT must be present; RBAC enforced at SecurityConfig; self-protection enforced in AdminService |
| AdminService → SessionRegistry | Server-side close — no client can trigger this directly; only reachable via ADMIN-authenticated endpoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Elevation of Privilege | /api/admin/users/{id}/lock — regular user calling admin endpoint | mitigate | SecurityConfig authorizeHttpRequests: /api/admin/** requires ADMIN role; enforced at filter chain before controller (per PATTERNS.md §Shared Security) |
| T-05-08 | Denial of Service | Admin accidentally locks own account | mitigate | adminService.lockUser checks adminUsername.equals(targetUsername) → throws IllegalArgumentException → 400; enforced server-side (D-10 authoritative). FE also disables own row but this is defense-in-depth only. |
| T-05-09 | Elevation of Privilege | Admin changes own role to USER (self-demotion) | mitigate | adminService.changeRole checks same self-protection condition → 400; matching the D-10 decision. |
| T-05-10 | Information Disclosure | Error message reveals internal state | mitigate | GlobalExceptionHandler returns structured ApiError (message, status, path) — no stack traces in production response. IllegalArgumentException message is admin-visible only (admin-only endpoint). |
| T-05-11 | Tampering | Role change request body with invalid role name | mitigate | Role.valueOf(roleName) throws IllegalArgumentException on unknown role → 400 via GlobalExceptionHandler. No unchecked enum access. |
| T-05-SC | Tampering | npm packages (no new packages in this plan) | accept | No new npm dependencies introduced; all existing packages already audited in prior phases. |
</threat_model>
