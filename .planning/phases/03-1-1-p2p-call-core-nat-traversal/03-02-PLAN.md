<objective>
Backend signaling + TURN vertical slice. Implement the opaque call-signaling relay on top of
the Phase 2 WebSocket layer and the ephemeral-HMAC TURN credential endpoint. After this plan, a
message sent by one authenticated client reaches the named target client — server-stamped `from`,
SDP/ICE passed through opaquely — and the browser can fetch time-limited TURN credentials.

This is the backend half of the core-value slice (CALL-01): the signaling channel that the
frontend PeerManager (plan 03) rides on. It also unblocks NAT traversal (INFR-01) by serving
ephemeral credentials.

Purpose: Implement the Phase 2 `sendToUser` stub and add call message types to the sealed
envelope without parsing media; serve TURN credentials per the TURN REST API spec.
Output: CallSignalingTest and TurnControllerTest go GREEN.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Spring WS handler | Untrusted client signaling messages cross here; `to` and SDP/ICE are attacker-controllable |
| browser → REST /api/turn-credentials | Authenticated request; principal is trusted (JWT filter), request params are not |
| Spring → coturn (shared secret) | turn.secret is server-only; never sent to the browser in static form |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | signaling `from` field | mitigate | Handler stamps `from = username(session)` from the authenticated principal; client `from` ignored (asserted by CallSignalingTest spoof case) |
| T-03-03 | Tampering | routing injection via `to` | mitigate | sendToUser only delivers to a session registered in SessionRegistry; an unknown/offline `to` is dropped+logged, never broadcast |
| T-03-02 | Information Disclosure | TURN credentials | mitigate | Ephemeral HMAC-SHA1 with TTL (24h default); static secret stays server-side in turn.secret; browser only receives time-limited derived credential |
| T-03-04 | Elevation of Privilege | TURN open relay | mitigate | coturn use-auth-secret (plan 05) validates the HMAC; credential is bound to authenticated principal in the username |
| T-03-05 | Tampering | opaque SDP/ICE passthrough | accept | Server never parses media payloads (by design); WebRTC ICE connectivity checks + DTLS fingerprint reject forged candidates at the peer |
| T-03-SC | Tampering | maven installs | accept | No new maven dependencies (javax.crypto is JDK stdlib) |
</threat_model>
