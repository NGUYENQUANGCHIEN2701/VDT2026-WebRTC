<objective>
Frontend core-value vertical slice — the working 1-1 P2P call. This is the slice the entire
product exists to protect: after this plan, two users on the same network can place a video/audio
call where media flows peer-to-peer. Built on the Phase 2 WS layer and the plan-02 signaling relay,
using STUN-first (D-03) so signaling correctness is provable before coturn is added (plan 05).

Perfect negotiation (callee always polite) + ICE candidate buffering ship in PeerManager's first
commit (CLAUDE.md mandate — not retrofit). The peer manager is a plain TS module: RTCPeerConnection
and MediaStream live in module scope, never in Zustand; only serializable derived state crosses into
callStore. Self-view is mirrored; getUserMedia failures degrade gracefully to audio-only or an
actionable Vietnamese error.

Purpose: Deliver the actual call experience (CALL-01, MEDIA-02, MEDIA-05) — call initiation, incoming
accept/reject, cancel, hang-up, and a connected media session over STUN.
Output: PeerManager.test.ts and media.test.ts go GREEN; a manual 2-tab call connects on localhost.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| remote peer → local RTCPeerConnection | SDP/ICE from the remote peer is untrusted; the browser ICE agent + DTLS handle validation |
| browser → camera/mic (getUserMedia) | Browser-enforced permission + secure-context gate |
| client UI → callStore | Only serializable state crosses; non-serializable media objects stay in PeerManager |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Tampering | forged remote ICE candidate | mitigate | Native ICE agent rejects candidates that fail connectivity checks; DTLS fingerprint verification prevents MITM (no app-layer SDP parsing) |
| T-03-07 | Spoofing | unauthorized call initiation | mitigate | call-offer rides the Phase-2 authenticated WS; the server stamps `from` (plan 02) so the callee always sees the true caller identity |
| T-03-08 | Information Disclosure | media object leak via devtools | mitigate | RTCPeerConnection/MediaStream kept out of Zustand (CLAUDE.md mandate); only derived strings in store |
| T-03-09 | Denial of Service | getUserMedia abuse / device hang | accept | Browser-controlled permission prompt; errors degrade to actionable notice / audio-only, not a crash |
| T-03-SC | Tampering | npm installs | accept | No new npm dependencies in this plan (native browser APIs only) |
</threat_model>
