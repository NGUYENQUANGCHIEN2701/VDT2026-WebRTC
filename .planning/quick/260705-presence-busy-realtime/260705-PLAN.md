<objective>
Fix: presence "IN_CALL" (busy) status does not update in realtime for other online users when a call starts — only appears after a manual page refresh (F5).

Root cause: `CallStateMachine.createCall()` sets the Redis `user-call:{userId}` keys (which drive `IN_CALL` in `RedisPresenceService.snapshot()`) as soon as `handleInvite()` creates a call, but `CallService.handleInvite()`'s `OK` branch never calls `presence.publishChanged()`. This is the missing counterpart to quick task 260702-3sd, which added `publishChanged()` to the five ENDED-transition branches (onRingTimeout, handleReject, handleCancel, handleHangUp, onGraceExpired) but explicitly deferred the "goes busy" side as out of scope. That deferred gap is exactly what the user is now reporting.

Purpose: third-party observers (any other connected user, not the caller/callee) must see both parties flip to IN_CALL the moment a call starts (ringing), without needing to reconnect or refresh.
Output: one-line fix in CallService.java + a unit test (mirroring CallServicePublishTest's existing pattern) + an integration test (mirroring CallLifecycleTest's existing third-party-observer pattern, symmetric to `hangup_notifiesThirdPartyObserverPresenceWithoutReconnect`).
</objective>
