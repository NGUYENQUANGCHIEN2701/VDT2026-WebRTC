<objective>
Sync the group-call screen-share "presentation mode" across ALL participants (not just the sharer's own browser tab), enforce a single-sharer-at-a-time rule authoritatively on the server, and make the recording compositor draw whichever participant is actually sharing.

Purpose: Today `isScreenSharing` is a local-only Zustand flag. When Alice shares her screen, only Alice's own UI flips into presentation layout — Bob and Carol still see the normal grid with Alice's shared content rendered as an ordinary small tile (or not distinguished at all), which does not match how screen sharing behaves in any real group-call product (Meet/Zoom) and breaks the "everyone sees what's being presented" expectation. The just-fixed recording compositor (quick task 260701-tkz) also still hardcodes the LOCAL stream into the presentation main/speaker regions unconditionally, so if a remote participant is the sharer, the recording would draw the wrong (or blank) stream once this fix ships.

Output:
- Backend: `MediaState`/`MediaStateRelay` carry an `isScreenSharing` flag; `PresenceWebSocketHandler` + `RoomService`/`RoomRepository` enforce a server-authoritative single-sharer-per-room lock (Redis-backed, atomic Lua claim, released on stop/leave/disconnect).
- Frontend state: `roomStore.RoomMember` tracks `isScreenSharing` per member; `roomActions.ts` transmits/receives it and gates `startRoomScreenShare()` against the synced room state.
- Frontend UI: `GroupCallPage.tsx`'s presentation-layout JSX renders whichever participant (self or a specific remote) is the actual sharer.
- Recording: `RecordingController` draws the actual sharer's video element into the main/speaker canvas regions, for both the local-is-sharer and remote-is-sharer cases.
</objective>
