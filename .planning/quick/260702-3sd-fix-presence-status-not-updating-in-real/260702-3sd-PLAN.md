<objective>
Fix a realtime presence bug: when a 1-1 call ends (missed/rejected/cancelled/completed/dropped), `transition_call.lua` deletes the Redis `user-call:{userId}` pointers that make `RedisPresenceService.snapshot()` report a user as ONLINE vs IN_CALL, but nothing publishes to the "presence-events" Redis pub/sub channel the way `RedisPresenceService.join()`/`.leave()` already do. Only the two call participants get a `CallStateChanged` message (which the frontend does not use to refresh presence status); every other connected client keeps showing the just-hung-up user as busy/in-call until something else forces a fresh snapshot (e.g. a page reload). This plan adds the missing publish to the Java call-lifecycle layer (mirroring the existing join/leave pattern, not the Lua script, for testability) and proves it with a new observer-side integration test.

Purpose: Presence status must be realtime and correct for every connected client, not just call participants — this is core to "see who's online" being trustworthy.
Output: `CallService` publishes a presence-events change on every ended-call transition; a new integration test proves a third-party observer converges without reconnecting.
</objective>
